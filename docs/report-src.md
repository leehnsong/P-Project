# 1. 문서 개요

본 보고서는 YOLO 영상분석 기반 스마트 주차 점유 감지 시스템의 전체 구조와, 이번 개발 주기에서 수행한 기능 고도화(지도 기반 주차장 동적 등록, 장소명 검색, 음성 질의응답, 안드로이드 앱 확장)의 내용을 처음부터 끝까지 정리한 것이다.

- 대상 독자: 프로젝트 팀원, 지도 교수/평가자
- 범위: 백엔드(Spring Boot, FastAPI), 웹 프런트엔드, 안드로이드 앱, 외부 연동(네이버, Gemini)
- 작성일: 2026-06-13

# 2. 프로젝트 개요

본 프로젝트는 드론/CCTV 영상에서 차량을 탐지하여 주차 구획(슬롯)의 점유 여부를 자동으로 판단하고, 그 현황을 웹과 모바일 앱에서 실시간으로 제공하는 스마트 주차 시스템이다.

핵심 가치는 다음과 같다.

- 영상 한 대로 다수의 주차 구획 점유를 자동 계산(수동 센서 불필요)
- 사용자가 지도에서 직접 주차장을 등록·관리(코드 수정 없이 운영 가능)
- 지도·검색·음성 등 다양한 방식으로 빈자리 확인

# 3. 추진 배경 및 목적

## 3.1 기존 한계

- 건물·주차장 정보가 소스코드에 하드코딩되어 있어 새 주차장 추가 시 코드 수정·재배포가 필요했다.
- 데이터가 인메모리 DB에 저장되어 서버를 재시작하면 모든 등록 정보가 사라졌다.
- 사용 방식이 웹 화면 조회에 한정되어 접근성이 낮았다.

## 3.2 목적

- 지도에서 위치를 지정해 주차장을 동적으로 등록·삭제할 수 있도록 개선한다.
- 등록 데이터를 영속 저장하여 재시작 후에도 유지한다.
- 장소명 검색, 음성 질의 등 사용 편의 기능을 추가한다.
- 안드로이드 앱으로 사용 범위를 모바일까지 확장한다.

# 4. 시스템 아키텍처

## 4.1 구성요소

| 구성요소 | 역할 | 주요 기술 |
|---|---|---|
| FastAPI 추론 서버 | 영상에서 차량 탐지, 슬롯 점유 계산, 현황 제공 | Python, YOLO(ultralytics), OpenCV |
| Spring Boot 서버 | 캠퍼스/건물/주차장 관리 API, 웹 UI, 인증, 음성·검색 프록시 | Java 17, Spring Boot 4, JPA, H2, Spring Security |
| 웹 프런트엔드 | 지도/현황 조회, 주차장 등록, 검색, 음성 | HTML/CSS/JS, 네이버 지도, Web Speech API |
| 안드로이드 앱 | 모바일 조회·음성 클라이언트 | Android(Java), Retrofit, 네이버 지도 SDK |
| 외부 연동 | 지도 표시, 장소 검색, 자연어 답변 생성 | 네이버 지도/지역검색, Google Gemini |

## 4.2 데이터 흐름

1. FastAPI가 `videos/` 폴더의 영상을 주기적으로 스캔(약 5초)하여 새 영상을 자동 인식한다.
2. YOLO가 영상 프레임에서 차량을 탐지하고, 슬롯 영역(slots.json)과 대조하여 점유 여부를 계산한다.
3. 계산 결과는 FastAPI `/status`로 제공되고, Spring Boot가 이를 주기적으로 폴링하여 캐시한다.
4. 웹/앱은 Spring Boot API를 통해 캠퍼스·건물·주차장 현황과 실시간 점유 상태를 받아 표시한다.
5. 음성 질의 시, Spring Boot가 현재 점유 현황을 요약하여 Gemini에 전달하고 자연어 답변을 생성해 반환한다.

## 4.3 데이터 모델(엔티티)

```
Campus 1 ── N Building 1 ── N ParkingLot
User 1 ── N SavedParkingLocation
User 1 ── N ParkingAlertRule
User 1 ── N InAppNotification
```

- **Campus**: 캠퍼스(지도 중심 좌표, 기본 줌)
- **Building**: 건물(이름, mapKey, 위도/경도). 지도 마커 단위
- **ParkingLot**: 주차장(이름, partitionKey, 슬롯 레이아웃). 점유 계산 단위
- **User / SavedParkingLocation / ParkingAlertRule / InAppNotification**: 로그인 사용자와 그 부가 데이터(주차 위치, 알림 규칙, 인앱 알림)

식별자 규칙

- `mapKey`: 건물 식별자(파일·마커 그룹 기준). 동적 생성 시 `bldg-xxxxxxxx` 형태
- `partitionKey`: 주차장 식별자(영상·이미지·슬롯·점유 상태 연결 기준). `{mapKey}_{번호}` 형태
- 파일 규칙: `videos/{partitionKey}_video.mp4`, `images/{partitionKey}_image.png`, `map/{partitionKey}_map.png`, `map/{partitionKey}_slots.json`

# 5. 구현 기능 상세

## 5.1 주차 점유 감지 (기반 기능)

- FastAPI가 영상을 읽어 일정 프레임 간격(기본 30프레임)마다 YOLO 추론을 수행한다.
- 탐지된 차량 중심점이 슬롯 영역 내부에 있으면 해당 슬롯을 점유로 판정한다.
- 슬롯 정의(slots.json)는 맵 빌더(OpenCV GUI)로 배경 사진 위에 주차 칸을 그려 생성한다.
- 결과는 주차장(partitionKey)별 총 칸 수, 빈자리 수, 장애인석 빈자리 수로 요약된다.

## 5.2 지도 기반 주차장 동적 등록 (신규)

기능 흐름

1. 웹 지도에서 빈 영역을 클릭한다.
2. 건물 이름을 입력하면 클릭 좌표로 **건물(Building)** 이 생성되고 지도에 마커가 즉시 표시된다.
3. 생성한 건물에 **영상을 업로드**하여 주차장(ParkingLot)을 추가한다. partitionKey가 자동 생성되고 영상이 서버에 저장된다.
4. FastAPI가 새 영상을 자동 인식(약 5초)하여 점유 계산을 시작한다.
5. 잘못 등록한 경우 건물/주차장을 삭제할 수 있으며, 연관 레코드와 파일이 함께 정리된다.

주요 개선

- 하드코딩되어 있던 건물 시드를 제거하고, 모든 건물·주차장을 런타임에 등록하도록 변경
- H2 데이터베이스를 인메모리에서 **파일 모드**로 전환하여 재시작 후에도 등록 데이터가 유지되도록 함
- 영상 업로드를 위해 업로드 용량 한도를 상향(2GB)하고 디스크 스트리밍 저장 적용

## 5.3 장소명 검색 (신규)

기능 흐름

1. 검색창에 "가천대"와 같은 장소명을 입력한다.
2. 서버가 네이버 지역검색 API(서버 프록시)를 호출하여 좌표를 받아 지도를 해당 위치로 이동한다.
3. 이동 후 사용자가 지도를 클릭하여 주차장을 등록할 수 있다.

비고

- 비밀키는 서버에만 보관(프록시 방식)하며, 무료 호출 한도 내에서 운용한다.

## 5.4 음성 질의응답 (신규)

기능 흐름

1. 사용자가 음성으로 "AI공학관 빈자리 있어?"와 같이 묻는다.
2. (앱) 안드로이드 네이티브 STT, (웹) 브라우저 Web Speech API가 음성을 텍스트로 변환한다.
3. 서버가 현재 전체 주차 현황을 요약하여 사용자 질문과 함께 Gemini(LLM)에 전달한다.
4. Gemini가 현황을 근거로 자연어 답변을 생성한다(현황에 없는 곳은 모른다고 답하도록 지시).
5. 답변을 화면에 표시하고 TTS로 음성 출력한다.

설계 요점

- 점유 수치는 시스템이 이미 계산하므로 LLM은 "의도 해석 + 자연어 답변 생성"만 담당한다.
- 모델은 경량·저비용의 `gemini-2.5-flash-lite`를 사용한다.
- 음성 입출력은 클라이언트(브라우저/안드로이드) 기본 기능을 사용하여 서버 비용을 최소화한다.

## 5.5 안드로이드 앱 (신규/확장)

- 홈 화면: "주차 현황 확인하기(지도)", "음성으로 질문하기" 진입
- 지도 화면: 서버의 건물 마커 표시 → 마커 클릭 시 그 건물의 **주차장(파티션)별 현황 카드**(이름 + 빈자리/총칸 + 상태 색상: 여유/혼잡/만차) 표시
- 음성 화면: 네이티브 STT → 서버 질의 → TTS 음성 답변(왕복 전체), 예시 안내 문구 제공
- 관리 전용 기능(맵 빌더)은 모바일에 부적합하여 앱에서 제외
- 백엔드 응답 형식 불일치(점유 갱신 시각 타입) 버그 수정

## 5.6 사용자/인증 및 부가 기능

- 인증: 회원가입/로그인, JWT 토큰 기반 인증
- 내 주차위치 저장: 슬롯 선택 후 현재 주차 위치 저장·해제(백엔드 API 구비)
- 빈자리 알림: 임계값 기반 알림 규칙 등록, 조건 충족 시 인앱 알림 생성(백엔드 API 구비)
- 조회·음성은 비로그인 허용, 주차위치·알림은 로그인 필요

# 8. 개발 환경 및 실행 방법

1. 백엔드 실행: 프로젝트 루트에서 통합 스크립트 실행
   - `./run.sh` (Spring Boot + FastAPI 동시 실행, 환경변수 자동 로드)
   - 웹: `http://localhost:8080/`
2. 앱 실행: Android Studio에서 에뮬레이터/실기기로 실행
   - 앱은 에뮬레이터 기준 `http://10.0.2.2:8080`로 백엔드를 호출
3. 음성·검색 기능은 백엔드에 해당 API 키(환경변수)가 설정되어 있어야 동작

# 9. 보안 및 키 관리

- 모든 외부 API 키(네이버 지도/지역검색, Gemini)와 JWT 비밀키는 **환경변수**로 분리 관리하며 소스에 포함하지 않는다.
- 검색·음성은 서버 프록시 방식으로 비밀키를 클라이언트에 노출하지 않는다.
- 데이터베이스 파일은 버전관리에서 제외한다.

# 10. 테스트 및 검증

- 백엔드 단위/통합 테스트(JUnit, MockMvc) 작성 및 전체 통과
- 신규 API(건물·주차장 등록/삭제, 음성, 검색)는 서버 기동 후 실제 호출로 동작 검증
- 데이터 영속성(재시작 후 등록 데이터 유지) 검증
- 앱은 빌드 검증 후 에뮬레이터에서 수동 동작 확인

# 12. 향후 계획 (로드맵)

- 1단계: 슬롯 단위 조회 화면, 주차장 즐겨찾기(자주 보는 주차장 빠른 확인)
- 2단계: 로그인(JWT) 연동, 내 주차위치 저장/추적(슬롯 선택 저장)
- 3단계: 빈자리 알림 설정 및 알림함
- 운영 단계: 데이터베이스 전환(H2 → MySQL/PostgreSQL), 푸시 알림(FCM) 도입 검토

# 13. 리스크 및 고려사항

- 점유 정확도는 슬롯 정의(맵 빌더) 품질과 영상 구도에 의존한다.
- 음성 인식 품질은 단말/브라우저에 따라 편차가 있다.
- 외부 API는 무료 한도가 있으므로 대규모 트래픽 시 유료 전환 또는 대안 검토가 필요하다.
- LLM 답변은 비결정적이므로 현황 데이터에 근거한 응답으로 제한한다.
